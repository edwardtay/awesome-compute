00 / START

Start with the workload, not the logo

“Compute” is too broad for one flat list. The list becomes useful when every category answers a real selection question.

I need to... Start with Choose when Watch out for
Run long-lived services Kubernetes / Nomad Mixed services, strong ecosystem, platform team Operational overhead and control-plane complexity
Schedule HPC jobs Slurm Batch, MPI, tightly managed clusters Cloud-native service patterns are not its core model
Run many independent jobs HTCondor High-throughput workloads on shared or opportunistic machines Not designed for tightly coupled low-latency jobs
Parallelize Python Ray / Dask Python-native tasks, actors, arrays, ML workloads Data movement and object-store pressure
Process very large data Spark / Flink Batch or stateful streaming across many nodes JVM tuning, shuffle cost, operational footprint
Serve LLM inference vLLM / SGLang GPU throughput, batching, caching, and structured output Model-specific optimization and compatibility churn
Run event-driven functions Knative / OpenFaaS Bursting HTTP/event workloads and scale-to-zero Cold starts and platform lock-in
Build reproducible pipelines Argo / Airflow / Nextflow DAGs, retries, artifacts, scheduled work A workflow engine is not a general compute scheduler
HARDWARE / QUICK MAP

Know what kind of compute you are buying

CPU and GPU are broad processor categories. TPU, LPU, and wafer-scale engines are vendor-specific architectures; NPU is an umbrella term used for many edge and client AI accelerators.

Type Use it for Key constraint
CPU Flexible application logic, databases, preprocessing, small or irregular models Lower tensor throughput than accelerators
GPU General accelerated computing, AI training and inference, simulation Memory, power, interconnect, and utilization
TPU Large regular ML workloads using Google Cloud and XLA Framework operations, shapes, and cloud availability
NPU Power-efficient local AI on client, mobile, and edge devices Vendor toolchains and weak cross-device portability
LPU Predictable, low-latency supported-model inference on Groq Vendor-specific inference platform
Wafer-scale Large-model training and high-speed inference on Cerebras Specialized service and software model
FPGA Custom data paths, deterministic latency, networking, streaming Specialized hardware development effort
ASIC Maximum efficiency for a stable, high-volume workload High design cost and low flexibility

DECISION / DISTRIBUTED DATA

Ray vs. Dask vs. Spark vs. Flink

They overlap at the edges, but their center of gravity is different: application runtime, Python collections, data platform, or stateful streaming.

Choose Programming model Strongest fit Avoid when
Ray Python tasks, actors, distributed objects AI systems and distributed application logic The workload is mainly SQL or dataframe ETL
Dask Python task graphs, NumPy, Pandas, Xarray Scaling existing Python analytics and science You need durable actors or a large SQL platform
Spark Dataframes, SQL, RDDs, structured streaming ETL, lakehouse, batch analytics, mature data platforms You need fine-grained Python actor communication
Flink Stateful dataflow, event time, checkpoints Continuously running, low-latency event processing The job is an ordinary Python task or periodic batch pipeline
02 / PROVIDERS

Commercial compute belongs in a separate comparison

Open-source projects and paid providers have different evidence. Provider entries should expose pricing model, regions, hardware, minimum commitment, egress, and whether the product is self-service.

Public hyperscale cloud

breadth

AWS, Google Cloud, Azure, Alibaba Cloud, and Oracle Cloud Infrastructure.

  • Deep service catalogs and global regions
  • Complex pricing, quotas, and egress
  • Choose for geography and ecosystem fit

GPU clouds

AI focus

CoreWeave, Lambda, Runpod, Crusoe, Vast.ai, and Prime Intellect.

  • Compare exact GPU SKU and interconnect
  • Separate on-demand from reserved
  • Record storage and egress cost

Serverless AI compute

low ops

Modal, Baseten, Replicate, Beam, Cerebras Inference, and GroqCloud.

  • Fastest path from code to endpoint
  • Measure latency, cold start, and concurrency
  • Distinguish model APIs from raw accelerator access

Marketplaces

variable supply

Vast.ai, Akash, Salad, and community capacity networks.

  • Can be materially cheaper
  • Availability and networking vary
  • Trust model must be explicit

Edge compute

latency

Cloudflare Workers, Fastly Compute, and Vercel Functions.

  • Put execution near users
  • Runtime and duration limits matter
  • Not a replacement for every backend

Bare metal

control

Equinix Metal, Hetzner, and dedicated providers.

  • Predictable performance
  • You own more operations
  • Excellent for stable high utilization
Operator is not provider: Meta and xAI run hyperscale AI infrastructure, but they do not sell general-purpose public compute. Their open-source systems and model APIs should be evaluated in those narrower categories.

03 / WHAT “BEST” MEANS

The strongest awesome lists are editorial products

A long README is not automatically useful. The best lists reduce a decision, state their boundaries, remove weak entries, and show signs of active stewardship.

1. Narrow promise

One audience, one topic, one clear definition of what is in and out. “Compute” needs explicit lanes; otherwise it collapses into cloud, DevOps, AI, and hosting.

2. Real curation

Reject duplicates, thin wrappers, abandoned experiments, SEO pages, and projects without a usable release. More entries should make selection easier, not harder.

3. Decision context

Every entry needs a differentiator. “Distributed computing framework” is weak; “Python actors and tasks with a shared object store” helps someone choose.

4. Visible maintenance

Automated link checks, activity health, contribution rules, a graveyard, and a recent editorial review date create trust.

5. Consistent schema

Name, official link, one-sentence description, license, language, activity, and best-for fields. Comparable records support tables and generated sites.

6. Community loop

Structured issue forms and PR checks make contributions cheap to review. Require the submitter to explain differentiation and evidence.

The key rule: use GitHub stars to describe adoption, never to decide quality alone. Stars reward age, brand, and launch distribution; curation must also judge maintenance, documentation, release quality, uniqueness, and practical use.
04 / APIs + AUTOMATION

How awesome lists get and maintain their data

Most good lists do not need a proprietary data API. They combine a curated source file with GitHub metadata, link validation, scheduled CI, and optional package-registry checks.

Source Use it for Do not use it for Recommended pattern
GitHub REST repo API Stars, forks, archive state, topics, license, language, pushed date Determining whether a tool is genuinely good Scheduled authenticated snapshot; cache JSON in the repo
GitHub GraphQL API Batch many repositories in fewer round trips Anonymous browser calls Use in GitHub Actions with the built-in token
Lychee Dead links, redirects, scheduled link health Proving a commercial product still works Weekly CI; open an issue or PR with failures
Shields endpoint badges README project count, review date, and snapshot freshness Fetching dozens of repositories every time a README renders Publish tiny badge JSON files from scheduled CI
Package registries Latest npm, PyPI, crates.io, Go module, Docker image version Cross-language popularity comparison Only query the registry relevant to each project
Curated YAML/JSON Category, best-for, caveats, editorial score, provider facts Fast-changing popularity metrics Keep this as the human-owned source of truth
Website HTML Rendering and interaction Canonical data storage Generate README and site from the same records

Recommended data flow

Curate `data/projects.yml`Humans own category, description, differentiator, caveat, and inclusion status.
Enrich on a scheduleGitHub Actions fetches metadata with `GITHUB_TOKEN`; never expose a token in browser JavaScript.
Write `data/snapshot.json`Store stars, activity, release, license, archived state, and the fetch timestamp.
Validate and generateSchema check, duplicate check, link check, then generate README tables and the resource site.
Open a review PRAutomation proposes changes; a human decides moves, removals, and description edits. 
# .github/workflows/refresh.yml
name: Refresh directory
on:
  schedule:
    - cron: "17 5 * * 1"
  workflow_dispatch:

permissions:
  contents: write
  pull-requests: write

steps:
  - uses: actions/checkout@v4
  - uses: actions/setup-node@v4
    with: { node-version: "22" }
  - run: npm ci
  - run: npm run validate
  - run: npm run refresh
    env:
      GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
  - run: npm run generate
  - uses: peter-evans/create-pull-request@v7
Badge rule: keep the README to three list-level badges at most: project count, editorial review date, and metadata snapshot age. The resource site can show richer statistics from the scheduled repository snapshot.
05 / INCLUSION SCORECARD

A repeatable standard for adding projects

Score evidence, then apply editorial judgment. A project should normally reach 70/100 and have no hard rejection flag.

Dimension Weight Full-credit evidence Hard warning
Practical utility 25 Solves a clear compute problem with usable docs or examples Only a paper, landing page, or thin wrapper
Maintenance 20 Recent meaningful commits/releases and responsive issue handling Archived, broken install, unexplained inactivity
Differentiation 15 Clear reason to choose it over adjacent entries Duplicate with no meaningful distinction
Adoption evidence 15 Users, contributors, integrations, deployments, citations, or ecosystem Stars are the only evidence
Documentation 10 Quickstart, architecture, deployment, and limitations No working getting-started path
Open governance 10 License, contribution path, releases, ownership No license or unclear code ownership
List fit 5 Fits one category and improves the reader's decision Added mainly for promotion or affiliate value
